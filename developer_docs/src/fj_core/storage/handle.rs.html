5/// # A handle that references a stored object
6///
7/// You can get an instance of `Handle` by inserting an object into a store. A
8/// handle dereferences to the object it points to, via its [`Deref`]
9/// implementation.
10///
11/// ## Bare objects and stored objects
12///
13/// A bare object is just that: an instance of a bare object type. Once a bare
14/// objects is inserted into storage, it becomes a stored object. A stored
15/// object is owned by the store, and can be referenced through instances of
16/// `Handle`.
17///
18/// The point of doing this, is to provide objects with a unique identity, via
19/// their location within storage. The importance of this is expanded upon in
20/// the next section.
21///
22/// ## Equality and Identity
23///
24/// Most objects have [`Eq`]/[`PartialEq`] implementations that can be used to
25/// determine equality. Those implementations are derived, meaning two objects
26/// are equal, if all of their fields are equal. This can be used to compare
27/// objects structurally. [`Handle`]'s own [`Eq`]/[`PartialEq`] implementations
28/// defer to those of the stored object it references.
29///
30/// However, that two objects are *equal* does not mean they are *identical*.
31///
32/// This distinction is relevant, because non-identical objects that are
33/// *supposed* to be equal can in fact end up equal, if they are created based
34/// on simple input data (as you might have in a unit test). But they might end
35/// up slightly different, if they are created based on complex input data (as
36/// you might have in a real-world scenario). This situation would most likely
37/// result in a bug that is not easily caught in testing.
38///
39/// You can compare the identity of two `Handle`s, by comparing the values
40/// returned by [`Handle::id`].
41///
42/// ### Validation Must Use Identity
43///
44/// To prevent situations where everything looks fine during development, but
45/// you end up with a bug in production, any validation code that compares
46/// objects and expects them to be the same, must do that comparison based on
47/// identity, not equality. That way, this problem can never happen, because we
48/// never expect non-identical objects to be equal.
49pub struct Handle<T> {
50    pub(super) store: StoreInner<T>,
51    pub(super) index: Index,
52    pub(super) ptr: *const Option<T>,
53}

55impl<T> Handle<T> {
56    /// Access the object's unique id
57    pub fn id(&self) -> ObjectId {
58        ObjectId::from_ptr(self.ptr)
59    }
60
61    /// Return a bare object, which is a clone of the referenced stored object
62    pub fn clone_object(&self) -> T
63    where
64        T: Clone,
65    {
66        self.deref().clone()
67    }
68}

70impl<T> Deref for Handle<T> {
71    type Target = T;
72
73    fn deref(&self) -> &Self::Target {
74        // `Handle` keeps a reference to `StoreInner`. Since that is an `Arc`
75        // under the hood, we know that as long as an instance of `Handle`
76        // exists, the `StoreInner` its data lives in is still alive. Even if
77        // the `Store` was dropped.
78        //
79        // The `Store` API ensures two things:
80        //
81        // 1. That no `Handle` is ever created, until the object it references
82        //    has at least been reserved.
83        // 2. That the memory objects live in is never deallocated.
84        //
85        // That means that as long as a `Handle` exists, the object it
86        // references has at least been reserved, and has not been deallocated.
87        //
88        // Given all this, we know that the following must be true:
89        //
90        // - The pointer is not null.
91        // - The pointer is properly aligned.
92        // - The pointer is dereferenceable.
93        // - The pointer points to an initialized instance of `T`.
94        //
95        // Further, there is no way to (safely) get a `&mut` reference to any
96        // object in a `Store`/`Block`. So we know that the aliasing rules for
97        // the reference we return here are enforced.
98        //
99        // Furthermore, all of the code mentioned here is covered by unit tests,
100        // which I've run successfully under Miri.
101        let slot = unsafe { &*self.ptr };
102
103        // Can only panic, if the object was reserved, but the reservation has
104        // never been completed.
105        slot.as_ref()
106            .expect("Handle references non-existing object")
107    }
108}

152impl<T> fmt::Debug for Handle<T>
153where
154    T: fmt::Debug,
155{
156    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
157        let name = {
158            let type_name = type_name::<T>();
159            match type_name.rsplit_once("::") {
160                Some((_, name)) => name,
161                None => type_name,
162            }
163        };
164        let id = self.id().0;
165        let object = self.deref();
166
167        if f.alternate() {
168            write!(f, "{name} @ {id:#x} => {object:#?}")?;
169        } else {
170            write!(f, "{name} @ {id:#x}")?;
171        }
172
173        Ok(())
174    }
175}

180/// The unique ID of a stored object
181///
182/// You can access a stored object's ID via [`Handle::id`]. Please refer to the
183/// documentation of [`Handle`] for an explanation of object identity.
184#[derive(Clone, Copy, Eq, PartialEq, Hash, Ord, PartialOrd)]
185pub struct ObjectId(pub(crate) u64);
186
187impl ObjectId {
188    pub(crate) fn from_ptr<T>(ptr: *const T) -> ObjectId {
189        Self(ptr as u64)
190    }
191}
192
193impl fmt::Debug for ObjectId {
194    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
195        let id = self.0;
196        write!(f, "object id {id:#x}")
197    }
